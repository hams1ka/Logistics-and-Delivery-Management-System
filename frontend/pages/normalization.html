4.1 — Pitfall Analysis
PITFALL
Before normalization, a naive Logistics_Raw table was designed with all information crammed into a single unnormalized relation. This table violates 1NF because it contains multi-valued attributes (multiple phone numbers in one cell), repeating groups (multiple vehicles per driver row), and non-atomic values (comma-separated lists).
Unnormalized Table — Logistics_Raw (Before 1NF)

Violations Identified
  • Phone_Numbers contains multiple values separated by commas — not atomic
  • Vehicle_Types is a repeating group — multiple vehicles per driver row
  • Order_IDs is a comma-separated list — violates atomicity rule
  • Locations contains multiple city names in one cell — not atomic
4.2.1 — Identify Dependencies
1NF
1NF Rule: Every column must contain atomic (indivisible) values. Each row must be unique. No repeating groups or arrays are allowed. A table is in 1NF if and only if every attribute contains only single, atomic values.
Driver_id → Driver_Name, Phone_no (single value only) ATOMIC
Vehicle_Id → Plate_Number, Vehicle_Type, Capacity, Availability_Status ATOMIC
Location_id → Address (single city), Contact_No (single number) ATOMIC
4.2.2 — Apply 1NF
AFTER 1NF
After applying 1NF, every multi-valued attribute is split into separate tables. Each table has a primary key and all attributes are atomic. The repeating groups (vehicles, orders, locations per driver) are moved to their own tables linked by foreign keys.

4.3.1 — Identify Partial Dependencies
2NF
2NF Rule: A table is in 2NF if it is in 1NF AND every non-key attribute is fully functionally dependent on the entire primary key. Partial dependency occurs only when a composite primary key exists — a non-key attribute depends on only part of the composite key.
Analysis — Assignment Table (Composite Key Risk)
The Assignment table was initially designed with a composite key (Delivery_id, Driver_id) before Assignment_id was introduced as a surrogate key. In that design, Start_time depended only on Delivery_id (partial dependency). After introducing Assignment_id as the sole primary key, this partial dependency is eliminated.
(Delivery_id, Driver_id) → Vehicle_id, Start_time PARTIAL DEP (old design)
Delivery_id → Start_time PARTIAL DEP (old design)
Assignment_id (PK) → Delivery_id, Driver_id, Vehicle_id, Start_time FULL DEP (current design)

4.4.1 — Identify Transitive Dependencies
3NF
3NF Rule: A table is in 3NF if it is in 2NF AND no non-key attribute is transitively dependent on the primary key. A transitive dependency occurs when: A → B → C (where A is the PK, B is a non-key attribute that determines C, another non-key attribute).
Transitive Dependency Analysis — Orders Table
A potential transitive dependency was identified: if Customer_Id in the Orders table stored customer details like Customer_Name and Customer_Address directly, then: Order_Id → Customer_Id → Customer_Name (transitive). This was resolved by keeping only Customer_Id in Orders and not embedding customer details inside the Orders table.
Order_Id → Customer_Id → Customer_Name (hypothetical) TRANSITIVE (avoided)
Order_Id → Oder_Date, Customer_Id (reference only) 3NF COMPLIANT ✓
Transitive Dependency Analysis — Service_History Table
Another potential case: if Vehicle_Type were stored directly in Service_History instead of referencing Vehicle_id, then: Record_id → Vehicle_Type → Capacity (transitive). By storing only Vehicle_id (FK) and fetching type/capacity from the Vehicle table via JOIN, this transitive dependency is eliminated.
Record_id → Vehicle_Type → Capacity TRANSITIVE (avoided)
Record_id → Vehicle_id (FK), Maintenance_Type, Service_Date 3NF COMPLIANT ✓

4.5.1 — Identify BCNF Violations
BCNF
BCNF Rule (Boyce-Codd Normal Form): A table is in BCNF if for every functional dependency X → Y, X must be a superkey (candidate key). BCNF is a stronger version of 3NF. A table can be in 3NF but still violate BCNF if a non-superkey attribute determines another attribute.
BCNF Analysis — Driver Table
Driver_id (PK / superkey) → Name, Phone_no, License_Number BCNF ✓
License_Number (candidate key) → Driver_id, Name, Phone_no BCNF ✓ (License is also a superkey)
BCNF Analysis — Vehicle Table
Vehicle_Id (PK) → Plate_Number, Vehicle_Type, Capacity, Availability_Status BCNF ✓
Plate_Number (candidate key) → Vehicle_Id, Vehicle_Type, Capacity BCNF ✓ (Plate is also a superkey)
BCNF Analysis — Assignment Table
Assignment_id (PK / superkey) → Delivery_id, Driver_id, Vehicle_id, Start_time BCNF ✓

4.6.1 — Identify Multi-Valued Dependencies
4NF
4NF Rule: A table is in 4NF if it is in BCNF AND has no non-trivial multi-valued dependencies. A multi-valued dependency (MVD) exists when one attribute in a table uniquely determines multiple independent sets of values of another attribute. Notation: A ↠ B (A multi-determines B).
Multi-Valued Dependency Analysis — Assignment Table
A potential MVD scenario: if a driver could be assigned to multiple vehicles independently of deliveries, we would have: Driver_id ↠ Vehicle_id and Driver_id ↠ Delivery_id independently. This was resolved by the Assignment table structure where Driver, Vehicle and Delivery are all linked together in a single assignment record — they are not independent multi-valued facts.
Driver_id ↠ Vehicle_id AND Driver_id ↠ Delivery_id MVD RISK (avoided)
Assignment_id → Driver_id + Vehicle_id + Delivery_id (all linked together) 4NF ✓ (no independent MVD)
Multi-Valued Dependency Analysis — Service_History Table
If a vehicle could have multiple independent maintenance types AND multiple independent service dates (not linked to each other), that would be an MVD: Vehicle_id ↠ Maintenance_Type and Vehicle_id ↠ Service_Date independently. In our design, each Service_History record has a specific (Maintenance_Type, Service_Date) pair — they are co-dependent, not independent. No MVD exists.
Vehicle_id ↠ Maintenance_Type AND Vehicle_id ↠ Service_Date MVD RISK (avoided)
Record_id (PK) → Vehicle_id, Maintenance_Type, Service_Date (linked pair) 4NF ✓

4.7.1 — Identify Join Dependencies
5NF
5NF Rule (Project-Join Normal Form): A table is in 5NF if it is in 4NF AND every join dependency is implied by the candidate keys. A join dependency means the table can only be reconstructed by joining all its projections — it cannot be losslessly decomposed further. 5NF deals with cases where data can only be accurately represented by the full combination of three or more attributes.
Join Dependency Analysis — Assignment Table
The most complex relation in our system is the Assignment table which links Driver, Vehicle, and Delivery together. The key question for 5NF is: can this table be split into binary projections (Driver↔Vehicle, Vehicle↔Delivery, Driver↔Delivery) and then re-joined to get the same result without spurious tuples?
Projection 1 : Driver_Vehicle (Driver_id, Vehicle_id) LOSSY alone
Projection 2 : Vehicle_Delivery (Vehicle_id, Delivery_id) LOSSY alone
Projection 3 : Driver_Delivery (Driver_id, Delivery_id) LOSSY alone
Join Dependency : * {Driver_id, Vehicle_id, Delivery_id} — must be stored together 5NF ✓ (implied by PK)

Final Result — All Normal Forms Satisfied: The Logistics & Delivery Operations Management System database satisfies all six normal forms (1NF through 5NF). The design uses surrogate primary keys to eliminate partial dependencies, separates entities into their own tables to remove transitive dependencies, links multi-entity facts through the Assignment table to prevent multi-valued dependency violations, and stores all complex relationships as single lossless records to satisfy the join dependency requirement of 5NF.
